🐝 HIVE — Documentación General de Arquitectura
Plataforma de Red Team Multi-Agente con Inteligencia de Enjambre
1. Visión General
Hive es un framework de Red Team que implementa una colonia de agentes autónomos especializados, comunicándose sin C2 centralizado, tomando decisiones por consenso y evolucionando sus tácticas mediante aprendizaje automático y competición interna.

Principio fundamental: La inteligencia ofensiva no reside en un operador ni en un servidor, sino que emerge de la interacción descentralizada entre agentes que colaboran, compiten y se adaptan como un superorganismo biológico.

2. Filosofía de Diseño
Principio	Implementación
Sin punto único de fallo	Todos los agentes son pares; no hay C2 jerárquico
Comunicación sigilosa	Memoria compartida anónima + estigmergia en sistema de archivos + mesh P2P
Adaptación continua	MARL, torneos darwinianos, mutación polimórfica
Evasión multicapa	Syscalls directas, stack spoofing, fileless, anti-análisis, ofuscación
Persistencia profunda	Bootkit UEFI, estigmergia, agentes durmientes, resurrección post-formateo
Control descentralizado	Consenso ponderado con reputación, smart contracts operativos

Canales de comunicación:

Shared Memory Arena: comunicación local entre agentes (0 puertos, 0 sockets)

Estigmergia: rastros en xattr/ADS de archivos legítimos

Smoke Signals: C2 camuflado como tráfico de CDN/cloud legítimo

WhisperNet: mesh P2P cifrada para sincronización entre hosts

Pheromone Trails: marcas en sistema de archivos compartiendo inteligencia táctica

5. Agentes Principales
5.1 Worker — Abeja Obrera Exploradora
Característica	Detalle
Archivo	agents/worker/
Rol	Percepción del entorno
ML	RandomForest 100% Rust (14 features → 3 clases: normal/backup/EDR)
Módulos	recon.rs, ml.rs, pheromone.rs
Capacidades:

Detección de 8 familias EDR (CrowdStrike, Defender, SentinelOne, CarbonBlack, Cylance, Symantec, McAfee, Elastic)

Enumeración de procesos, interfaces de red, OS, usuario, shares

Clasificación del entorno con modelo ONNX/RandomForest

Publicación de creencias en la arena compartida

Escaneo de subredes vía ARP/nmap

Marcado de rastros de feromonas en el sistema de archivos

5.2 Drone — Zángano Estratega
Característica	Detalle
Archivo	agents/drone/
Rol	Toma de decisiones y regeneración
ML	Política MARL (62-dim → 10 acciones)
Módulos	consensus.rs, lateral.rs, regenerate.rs
Capacidades:

Recepción de creencias del Worker

Evaluación de política MARL para decidir acciones

Generación de propuestas y votación en consenso

Regeneración de agentes caídos desde RAM (fileless)

Movimiento lateral vía SSH/SCP

Orquestación de explotación controlada

Recuperación de conocimiento desde estigmergia

5.3 Honeybee — Abeja Recolectora
Característica	Detalle
Archivo	agents/honeybee/
Rol	Ejecución final
Módulos	encrypt.rs, exfil.rs, wipe.rs
Capacidades:

Cifrado de archivos con AES-256-GCM (clave aleatoria por sesión)

Exfiltración HTTP POST al C2 con chunking paralelo

Wipe seguro de 3 pasadas + zeros + unlink

Activación solo por consenso (umbral 80%)

Respeto de safe_mode y safe_ips

Publicación de resultados en arena compartida

5.4 Weaver — Abeja Tejedora
Característica	Detalle
Archivo	agents/weaver/
Rol	Ofuscación y mutación
IA	LLM local (Ollama) + 4 técnicas deterministas
Módulos	mutate.rs, obfuscate.rs, lolbin_chain.rs
Técnicas de mutación:

XOR multi-byte con keystream aleatorio

Inserción de NOP sleds de tamaño variable

Reordenamiento de secciones del binario

Inyección de código basura (dead code)

Capacidades adicionales:

Generación de cadenas LOLBin ofuscadas

Stealth download + exec vía LOLBins

Mutación de payloads de exploits

Generación de variantes para torneos

5.5 Queen — Reina
Característica	Detalle
Archivo	agents/queen/
Rol	Estrategia global y C2 bridge
Módulos	royal_jelly.rs, c2_bridge.rs, tournament.rs
Capacidades:

Emisión de directivas Royal Jelly (cambio de estrategia global)

Puente C2: traducción Sliver/CS/HTTP ↔ LdC

Orquestación de torneos darwinianos

Evaluación y scoring de competidores

Propagación del genoma ganador

Arbitraje de disputas de consenso

6. Agentes Especializados
6.1 Leech — Abeja Chupasangre
Cosecha credenciales de múltiples fuentes:

Kerberos: TGT/TGS del ticket cache

Git: .git-credentials, tokens en repositorios

Procesos: variables de entorno (PASSWORD, SECRET, TOKEN, KEY)

Navegadores: Chrome Login Data, Firefox logins.json

RDP: configuraciones FreeRDP

SSH: claves privadas en ~/.ssh/

6.2 Guardian — Abeja Guardiana
Defensa activa contra trampas del Blue Team:

Detección de honeyfiles (señuelos)

Detección de honeypots (patrones de red)

Detección de canary tokens

Verificación de integridad del entorno

6.3 Stinger — Fábrica de Payloads
Generación de stagers multi-formato:

Python one-liner

PowerShell script

wget + pipe + bash

curl + bash

Paquetes .deb/.rpm

6.4 Swarm — Enjambre Autónomo
Propagación sin consenso:

Auto-limitado: max 10 hops, 2 infecciones/min, 1h de vida

SSH key auth + SCP deploy automático

Respeto de safe_ips

Lectura de creencias del Worker para evitar EDRs

9. Sistema de Consenso
Parámetro	Valor por defecto	Descripción
Umbral de consenso	66%	Porcentaje de votos ponderados necesario
Peso inicial	1.0	Reputación inicial de cada agente
Recompensa	+0.1	Incremento por acierto
Penalización	-0.2	Decremento por fallo
Decay	0.001/minuto	Decaimiento hacia 1.0
Timeout de votación	30s	Tiempo máximo para alcanzar consenso

11. Ciclo de Vida de una Operación
text
FASE 1: DESPLIEGUE
  Stinger → memfd_create(agentes) → exec → autodestrucción
  Todos los agentes en RAM, cero disco

FASE 2: RECONOCIMIENTO
  Worker escanea sistema → ML clasifica → publica creencias
  Leech cosecha credenciales → deposita en arena

FASE 3: PLANIFICACIÓN
  Drone evalúa política MARL → genera propuestas
  Consenso ponderado con reputación → decisión colectiva

FASE 4: PROPAGACIÓN
  Drone despliega agentes vía SSH/SCP
  Swarm propaga autónomamente (si está activo)
  Estigmergia marca hosts comprometidos

FASE 5: ACCIÓN
  Honeybee cifra/exfiltra/destruye (si consenso ≥ 80%)
  Saboteadoras alteran datos (si RoyalJelly activo)

FASE 6: SUPERVIVENCIA
  Guardian detecta amenazas → alerta a la colonia
  Drone regenera agentes caídos
  Hibernación/reactivación por eventos
  Bootkit UEFI asegura persistencia profunda

FASE 7: EVOLUCIÓN
  Torneos darwinianos mejoran las tácticas
  Death Dance A/B testing de variantes
  Propagación del genoma ganador

15. Comandos de Operación
bash
# Build completo
./hive.sh build

# Modo operador (solo C2 + dashboard)
./hive.sh brain

# Modo colonia (C2 + agentes locales)
./hive.sh colony

# Modo desarrollo (agentes en terminal, sin C2)
./hive.sh dev

# Desplegar en víctima remota
./hive.sh deploy 192.168.1.50

# Ver estado de la colonia
./hive.sh status

# Detener toda la colonia
./hive.sh stop

# Ejecutar tests
./hive.sh test

# Validación EDR
python3 tests/validate_edr.py --output edr_report.json --html

# Dashboard
python3 tests/dashboard.py --port 8080

# Pipeline CI/CD
bash tests/edr_gauntlet.sh --watch --report results.json
